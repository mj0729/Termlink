## 1. 需求

### 背景
上一轮已将 `RemoteFileTable` 中的表头高度变量从 `28px` 下调到 `24px`，但用户截图反馈红框区域肉眼几乎无变化。进一步排查确认，真正占高的并不只是本地 `--remote-table-header-height`，`vxe-table` 默认的 `.vxe-table--render-default .vxe-header--column.is--padding .vxe-cell` 仍通过更高优先级的上下 padding 撑高了表头。

### 目标
- 修正远程文件表格表头的真实生效层级，让高度压缩在界面上实际可见。
- 保持路径栏、搜索栏、正文行高与排序/列拖拽交互不变。
- 将本次根因沉淀到知识库，避免后续继续只改低优先级变量。

### 约束条件
```yaml
时间约束: 本轮直接完成
性能约束: 不增加运行时逻辑或依赖
兼容性约束: 保持 vxe-table 表头排序、列宽拖拽和现有密度模式可用
业务约束: 仅修正文件表头的真实占高，不调整路径栏、搜索栏和正文行高
```

### 验收标准
- [ ] 表头实际视觉高度较当前版本明显收紧，不再出现“改了变量但肉眼没变化”
- [ ] 默认 `vxe-table` header padding 被本地样式正确覆盖
- [ ] `pnpm run build` 通过，且表头排序与列拖拽热区仍可用

---

## 2. 方案

### 技术方案
继续保留 `--remote-table-header-height` 作为表头目标高度，但把样式命中层上提到 `vxe-table` 默认渲染链：显式覆盖 `.vxe-table--render-default .vxe-header--column.is--padding .vxe-cell` 的上下 padding，并同步约束 header row / column / cell 的高度与 box model，使表头压缩在实际 DOM 上生效。

### 影响范围
```yaml
涉及模块:
  - ui-components: 远程文件工作台文件表格头部样式命中层修正
预计变更文件: 1
```

### 风险评估
| 风险 | 等级 | 应对 |
|------|------|------|
| 过度覆盖 vxe-table 默认样式导致表头交互区域错位 | 低 | 只覆盖 header padding 与高度，不改排序图标结构 |
| 调整后正文行高被连带影响 | 低 | 仅命中 `.vxe-header--column` 链路，不修改 body padding 规则 |

## 5. 技术决策

> 本方案涉及的技术决策，归档后成为决策的唯一完整记录

### remote-file-table-header-effective-height-fix#D001: 覆盖 vxe-table 默认 header padding，而不再只改低优先级高度变量
**日期**: 2026-03-23
**状态**: ✅采纳
**背景**: 用户确认上一轮视觉上“几乎没变化”，说明根因不在变量值本身，而在实际占高层未被命中。
**选项分析**:
| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 继续下调 `--remote-table-header-height` 数值 | 实现成本低 | 仍可能被更高优先级 padding 覆盖，重复无效调整 |
| B: 直接覆盖 `.vxe-table--render-default .vxe-header--column.is--padding .vxe-cell` | 直击真实占高层，结果可预期 | 需要更谨慎控制选择器范围 |
**决策**: 选择方案 B
**理由**: 这次问题的核心是命中层级错误，不是数值不够小。只有覆盖默认 header padding，才能让表头真实收紧。
**影响**: 影响 `RemoteFileTable.vue` 的表头样式策略，并更新知识库中关于远程文件表格头部紧凑化的说明
